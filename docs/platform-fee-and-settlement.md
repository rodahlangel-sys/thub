# 平台信息服务费与大学生结算

## 业务规则

- 家长支付订单总额的 100%，平台不会在订单总额之外额外收取 5%。
- 平台信息服务费固定为订单总额的 5%，使用 500 基点保存。
- 大学生预计结算金额为订单总额减去平台信息服务费。
- 家长确认辅导完成前，页面展示的费用均为预计值，不构成已结算收入。

## 金额计算

订单、支付、退款和结算金额统一使用整数分。大学生报价仍以元/小时维护，创建订单时由服务端转换为分并保存价格快照。

```text
平台信息服务费 = round(订单总额分值 × 500 / 10000)
大学生结算金额 = 订单总额分值 - 平台信息服务费
```

客户端不能提交或修改费率、信息服务费和大学生结算金额。订单金额、Payment 金额与 Settlement 金额都由服务端数据生成。

## 数据模型

Order 保存：

- `platformFeeRateBps`
- `platformFeeAmountFen`
- `tutorNetAmountFen`

Settlement 保存：

- 唯一 `orderId`
- 对应大学生 `tutorId`
- 订单总额、费率、平台信息费和大学生结算额快照
- MOCK 渠道、状态、模拟流水号与结算时间

一个订单最多存在一条 Settlement，数据库唯一约束用于阻止并发重复结算。

## 触发流程

1. 家长创建预约，服务端根据真实报价与课时计算订单金额和费用快照。
2. 老师确认后，家长支付完整订单总额，款项进入现有担保状态。
3. 大学生开始服务并提交课后反馈。
4. 家长确认完成时，服务端在同一事务中校验订单、支付、反馈、退款和费用快照。
5. 订单更新为 `COMPLETED`，创建状态为 `SETTLED` 的 MOCK Settlement，并发送一条结算通知。

事务失败时订单、结算和通知均不会部分写入。

## 模拟结算

当前 `SETTLEMENT_PROVIDER` 只允许 `MOCK`。内部流水号以 `MOCK-SETTLEMENT-` 开头，不代表银行、支付宝或微信真实转账。用户页面使用“结算完成”和“当前为平台模拟结算记录”，不使用“真实到账”等表述。

## 退款边界

- 结算前退款沿用现有退款流程，不创建 Settlement，也不确认平台信息费收入。
- `REFUND_REQUESTED` 状态不能确认订单完成。
- 已存在 `SETTLED` Settlement 的订单不能进入普通退款审批路径，需要管理员人工售后处理。
- 当前版本不实现真实资金追回、分账回退或 Settlement 自动冲正。

## 幂等与并发

- 家长确认完成时重新读取最新订单状态。
- 订单必须处于 `PENDING_PARENT_CONFIRM`，Payment 必须为 `PAID` 且金额完全一致。
- Settlement 的 `orderId` 唯一。
- 事务内使用状态条件更新，唯一冲突会返回自然错误。
- 同一订单不会重复创建结算或重复发送结算通知。

## 历史数据检查

运行：

```bash
npm run check:settlement-consistency
```

该命令默认只读，检查已完成但无结算、状态不一致、金额不一致、已退款仍结算以及 Payment 金额异常。只有显式运行以下命令才会为符合条件的历史订单创建 MOCK 结算：

```bash
npm run repair:settlement-history
```

修复模式不会处理已退款、支付不一致或金额异常订单。

## 页面展示

- 家长支付页：订单金额、本次支付及信息费不会额外增加支付金额的说明。
- 家长订单详情：支付金额和平台是否完成本单结算。
- 大学生订单详情：预计或实际信息费、预计或实际结算额、结算状态。
- 管理员结算记录：总额、费率、信息费、家教结算额、渠道、状态、时间和模拟流水号。

## 当前限制

- 仅记录平台内部模拟结算，不接入真实收款账户或打款渠道。
- 历史已完成订单不会自动生成 Settlement，需要管理员确认后显式运行修复命令。
- 金额字段仍沿用现有整数类型，未增加多币种支持。
- 已结算订单的售后回退需要人工处理，尚未实现真实资金冲正。
